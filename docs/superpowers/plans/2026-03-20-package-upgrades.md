# Package Upgrades Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Upgrade all outdated npm packages, prioritizing Prisma 6→7, then zod 3→4, then Next 15→16.

**Architecture:** Upgrade packages one major version at a time, verify with typecheck + build after each, commit between upgrades. No test suite exists — typecheck and build are the verification gates. Because each task produces a separate commit, you can `git revert` any task individually if a later upgrade proves too complex to fix quickly; partial upgrades (e.g., only Prisma + zod, deferring Next 16) are perfectly valid.

**Tech Stack:** Next.js 15→16, Prisma 6→7, zod 3→4, @t3-oss/env-nextjs 0.12→0.13, @types/node 20→25, TypeScript 5.8, tRPC v11, PostgreSQL

## Task 4: Upgrade zod 3 → 4

**Files:**
- Modify: `package.json`
- Possibly modify: `src/server/api/trpc.ts` — `ZodError` usage
- Possibly modify: `src/server/api/routers/*.ts` — if any deprecated validators used
- Possibly modify: `src/env.js` — zod schemas; `z.string().url()` is stricter in v4

**Background:** zod v4 has breaking changes. Key ones relevant to this codebase:

1. **`ZodError.flatten()`** — return shape changed. `.fieldErrors` values are now `string[]` instead of `string[] | undefined`. The code in `trpc.ts` calls `error.cause.flatten()` — TypeScript types may change but this is not a runtime crash.
2. **`ZodError` import** — still `import { ZodError } from "zod"`. No change needed.
3. **`z.string().url()`** — validation is stricter in v4. `src/env.js` uses `DATABASE_URL: z.string().url()`. A valid `postgresql://` URL will pass, but test this during `npm run build` since env validation runs at build time.
4. **`z.number().positive()` and `z.number().int()`** — still work, but default error messages changed. This is a runtime behavior change (error strings returned to tRPC callers), not a typecheck failure.
5. **`z.string().nonempty()`** — removed in v4; this codebase already uses `.min(1)` throughout, so no change needed.

Check the full zod v4 migration guide: https://zod.dev/v4

- [ ] **Step 1: Update zod in package.json**

  In `package.json` dependencies:
  ```json
  "zod": "^4.3.6"
  ```

- [ ] **Step 2: Install**

  ```bash
  npm install
  ```

- [ ] **Step 3: Typecheck**

  ```bash
  npm run typecheck
  ```

  The most likely error location is `src/server/api/trpc.ts` around the `zodError` formatter:
  ```ts
  // Current (line 49):
  zodError: error.cause instanceof ZodError ? error.cause.flatten() : null,
  ```
  In zod v4, `ZodError` class is still named `ZodError` and `instanceof` still works. If typecheck fails here, the fix is to import `ZodError` directly:
  ```ts
  import { ZodError } from "zod";
  ```
  (already present — check if the import changed in v4.)

  For any router file errors: `z.object()`, `z.string()`, `z.number()` are all unchanged in v4.

- [ ] **Step 4: Build (validates env.js URL validation at build time)**

  ```bash
  npm run build
  ```

  The build runs `src/env.js` which calls `z.string().url()` on `DATABASE_URL`. Set a dummy URL to test:
  ```bash
  DATABASE_URL=postgresql://user:pass@localhost:5432/db npm run build
  ```
  If this fails with a URL validation error, `z.string().url()` in zod v4 is rejecting the format — replace with `z.string().min(1)` in `src/env.js` for `DATABASE_URL` if needed.

- [ ] **Step 5: Commit**

  ```bash

  git commit -m "chore: upgrade zod 3 → 4, fix any breaking API changes"
  ```
